Data quality worksheet

Good AI starts with data you can explain.

Training data quality is not only a technical problem. It is a trust problem. This checklist helps readers evaluate whether a dataset is relevant, lawful, documented, and fair enough for the task.

Data quality scorecard

AreaAsk thisWhy it matters
RelevanceDoes the data match the real task?Wrong examples teach the model the wrong pattern.
PermissionCan this data be used for this purpose?Legal and ethical limits affect whether a model should be trained.
FreshnessIs the data current enough?Old examples can make results outdated.
CoverageWho or what is missing?Missing groups can create poor or unfair performance.
LabelsAre labels consistent?Conflicting labels create confusing teaching signals.
TestingIs there a separate test set?A model can look good on familiar examples and fail on new ones.

Practical review steps

  1. Write the exact task the model is supposed to support.
  2. List every data source and who controls it.
  3. Check for duplicates, missing fields, inconsistent formats, and impossible values.
  4. Review a small sample manually before trusting the whole dataset.
  5. Document what was removed, changed, or excluded.
  6. Keep training, validation, and test examples separate.
  7. Recheck results for groups, cases, or situations that may be underrepresented.

Example: customer support data

A support chatbot trained only on solved tickets may learn neat answers but miss unresolved complaints, unusual cases, or policy exceptions. A better dataset includes representative questions, correct resolutions, edge cases, and clear labels for when the bot should hand off to a person.

Common data problems to look for

  • Duplicate examples: The model may overlearn repeated cases and look more accurate than it really is.
  • Missing fields: Empty dates, locations, categories, or labels can hide important patterns.
  • Mixed meanings: One label may be used differently by different people or teams.
  • Outdated examples: Old policies, prices, terminology, or user behavior can weaken current results.
  • Uneven coverage: Some groups, languages, regions, or edge cases may be barely represented.
  • Permission gaps: Data may be available but not appropriate to use for training or publication.

Sample review note

"This dataset appears relevant to the task because it contains recent examples from the workflow we want to support. Before training, we still need to remove duplicates, confirm permission, document excluded records, review label consistency, and reserve a separate test set. We should also check whether the data covers the users and cases that the final system is expected to handle."

How this checklist connects to model trust

A model can fail even when the algorithm is strong if the data is weak. Clean data makes patterns easier to learn. Representative data makes performance more realistic. Clear labels make the target behavior easier to measure. Separate test data helps reveal whether the model can handle new examples instead of only memorizing familiar ones.
